## CRITICAL

1. **[clarity] timing-based 检测的停止条件公式中 `prev_S` 未定义**
   `Phase 0.2 Detect Parallel Slots`
   Fix: 明确定义 `prev_S` 的含义，例如改为 `prev_n`（上一轮并发数）并给出完整的伪代码示例，包括每轮 n 和判定阈值的具体计算

## MAJOR

1. **[consistency] Phase 1 生成多 concurrency level，但 Phase 2 伪代码缺少内层循环**
   `Phase 1 / Concurrency-Requests Calculation & Phase 2 Execution`
   Fix: 在 Phase 2 的执行伪代码中增加内层循环，明确每个 group 依次执行多个 concurrency level，并说明每个 level 是独立调用 Orchestrator 还是一次配置多 level

2. **[clarity] thinking mode 的 system prompt 检测方式可能误判**
   `Phase 0.3 Detect Thinking Mode`
   Fix: 明确 system prompt 方式的请求参数格式，并说明如何区分「模型真正进入 thinking mode」和「模型恰好在输出中包含 `<think>` 标签」的情况，或者直接移除这个不可靠的检测方式

3. **[completeness] evalscope engine 的配置、调用方式和结果映射未说明**
   `Phase 1 Config Generation / Phase 2 Execution`
   Fix: 补充 evalscope engine 的配置生成细节、调用方式、concurrency 控制机制，以及结果到统一 metrics 结构的映射逻辑

4. **[completeness] Dimension Analysis 未定义 group 失败时的聚合策略**
   `Phase 3 Comparison Report`
   Fix: 明确 group 失败时的聚合策略，例如：仅对 status=completed 的 group 计算平均值，如果某个维度对比中所有 group 都失败则显示 N/A

5. **[architecture-quality] benchmark.py 承担 4 个职责（probe/plan/execute/report），缺少内部模块化说明**
   `File Structure`
   Fix: 说明 benchmark.py 内部的模块化结构（如 class 划分或子模块），或将 report 生成逻辑拆分到独立模块

6. **[clarity] per-group timeout 600s 与 thinking+longalpaca 场景下的实际工作量不匹配**
   `Phase 2 Safety Mechanisms`
   Fix: 根据 thinking mode + longalpaca 场景下的预期单请求延迟，重新计算合理的 per-group timeout，或改为 per-level timeout

## MINOR

1. **[consistency] CLI 参数名 `--max-concurrency` 与 Edge Cases 中 `--parallel 1` 不一致**
   `CLI Interface`
   Fix: 统一为 `--max-concurrency`，将 Edge Cases 中的 `--parallel 1` 改为 `S=1`（detected parallel slots = 1）
