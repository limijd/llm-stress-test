## MAJOR

1. **[architecture-quality] 双引擎（evalscope + 自研）的设计缺少选型依据和 trade-off 分析**
   `2. 架构 — 分层插件架构`
   为什么需要两个引擎？各自的优势和适用场景是什么？什么情况下推荐用哪个？如果自研引擎功能完备，evalscope 引擎的长期存在意义是什么？
   Fix: 增加引擎选型对比小节，说明：1) evalscope 引擎的优势（如社区验证、内置数据集）；2) 自研引擎的优势（如更精细的指标控制、无外部依赖）；3) 推荐的使用场景；4) 长期维护策略

2. **[clarity] concurrency 和 requests_per_level 要求按索引一一对应，但未说明两个数组长度不一致时的行为**
   `3. 配置体系`
   AI 实现时会不确定是报错、截断还是用默认值填充。
   Fix: 明确校验规则：两数组长度必须一致，不一致时在 validate 阶段报错并给出修复建议

3. **[completeness] EvalScope 引擎的 run() 需要解析 evalscope perf CLI 的输出并转为 LevelResult，但 spec 未定义解析规则**
   `4. 引擎抽象与双内核 — EvalScope 引擎`
   哪些字段对应 RequestMetric 的哪些字段、输出是 JSON/CSV/stdout、如何提取每请求级别的 TTFT/TPOT/output_tokens 等均未说明。
   Fix: 增加 evalscope perf 输出格式说明小节，明确：1) evalscope 输出的格式；2) 关键字段到 RequestMetric 的映射关系；3) evalscope 不输出某些字段时的处理策略

4. **[completeness] 自研引擎非流式模式下 token 计数来源未定义**
   `4. 引擎抽象与双内核 — 自研引擎`
   非流式响应中 token 计数依赖 API 返回的 usage 字段，若 API 不返回 usage 则无法统计，spec 未定义此 fallback。
   Fix: 说明非流式模式下 token 计数的来源：1) 优先使用响应中的 usage.completion_tokens / usage.prompt_tokens；2) 若 API 未返回 usage 字段时的处理策略

5. **[completeness] 主流程"最高并发通过→生成报告，结束"的判定逻辑不明确**
   `5. 编排层与降级策略 — 主流程`
   是仅检查最高并发级别，还是要求所有级别都通过？如果并发梯度 [1,5,10,20,50] 中 50 通过但 10 失败，算通过还是不通过？
   Fix: 明确判定逻辑：是"最高并发级别通过即整体通过"还是"所有级别都必须通过"，以及中间级别失败时是否继续执行后续更高并发级别

6. **[clarity] 降级探测的请求数缩放取整规则未明确**
   `5. 编排层与降级策略 — 降级流程`
   未说明缩放后的取整规则（向上/向下/四舍五入），也未说明缩放结果为 0 或极小值时的处理。
   Fix: 明确缩放公式（如 requests = max(min_requests, round(base_requests * new_concurrency / base_concurrency))）和最小请求数下限

7. **[clarity] "连续 3 个请求网络不通→终止整个测试"中的"连续"定义不明确**
   `9. 错误处理与健壮性`
   是同一轮内连续 3 个，还是跨轮累计？并发场景下'连续'的判定需要明确。
   Fix: 明确连续失败的判定范围（同一轮内）和并发场景下的计数方式（如累计网络错误占比超过阈值）
